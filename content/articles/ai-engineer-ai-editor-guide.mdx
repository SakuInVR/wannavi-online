AIエンジニアを目指すなら、AIエディタはかなり強い味方になります。
ただし、最初に勘違いしやすいのは「AIエディタを入れたら、あとは勝手に作れる」という期待です。

Cursorの初心者向け実演例やGitHub Copilotとの比較を整理すると、AIエディタ選びで大事なのは「どれが最強か」ではありません。
作る、聞く、確認する、戻す。この4つを安全に回せるかです。

## 最初に見るべきは「計画を出せるか」

Cursorの実演で分かりやすかったのは、いきなりコードを書かせるのではなく、先にPlanで作業手順を出していたところです。
たとえば小さなゲームを作る場合でも、まずToDoやファイル構成をMarkdownで作らせ、それを読んでからBuildへ進む。

初心者ほど、この一手間が大事です。
AIが書いたコードは速いですが、方向がズレたまま速く進むと、あとで直す量も増えます。

見るべき機能は、派手な生成力よりこのあたりです。

| 機能 | 初心者に効く理由 |
| --- | --- |
| 計画モード | いきなり実装せず、作る順番を確認できる |
| Askモード | ファイルを書き換えずに質問できる |
| Agentモード | 複数ファイルの修正を任せられる |
| Diff確認 | AIが何を変えたか見てから採用できる |
| Git連携 | 壊れたときに戻せる |

特に大事なのは、AskとAgentを分けることです。
「このエラーの意味を教えて」はAskでいいです。
「このフォームを送信できるようにして」はAgentでいいです。

質問したいだけなのにAgentで大きく直させると、初心者はどこが変わったか追えなくなります。

## AIにプロジェクト全体を見せる

AIエディタを使っても見当違いな答えが返ってくるとき、多くの場合はAIが文脈を持っていません。
Cursor活用例では、`@codebase` でプロジェクト全体を参照させたり、`@src` のようにフォルダを絞って質問したりしていました。

これはかなり実践的です。
Next.jsのエラーは、いま開いている1ファイルだけが原因とは限りません。
コンポーネント、API route、型定義、環境変数、CSSのどこかが絡みます。

だから質問の仕方はこう変えます。

| 悪い聞き方 | 良い聞き方 |
| --- | --- |
| このエラー直して | `@codebase` を見て、このエラーの原因候補を3つ出して |
| ログイン作って | `@src/app` と `@src/lib` を見て、既存設計に合わせた実装計画を出して |
| Stripeのコード書いて | 公式ドキュメントURLを読んで、最新APIに合わせた実装案を出して |

AIの回答が古いと感じたら、公式ドキュメントを渡すのも有効です。
特にAI系や決済系のライブラリは更新が速いので、AIの学習済み知識だけに頼ると古い書き方が出ることがあります。

## CursorかCopilotかで悩みすぎない

比較して面白いのは、「CursorかCopilotか」の勝者を決めるより、作業で使い分けていたところです。
コードを書くメインはCursor、インライン補完やレビューにはGitHub Copilot、という分業です。

初心者は最初から両方に課金しなくてOKです。
でも、選び方の基準は持っておくと迷いにくいです。

| やりたいこと | 向いている使い方 |
| --- | --- |
| ゼロから小さなアプリを作りたい | Cursorのチャット、Plan、Agent |
| 既存コードを書き進めたい | Copilotの補完 |
| 変更内容を別視点で見たい | PRレビューや別AIでの確認 |
| 学習しながら進めたい | 生成より、解説と差分確認を重視 |

最初の1本は、Cursor系のAIエディタだけで十分です。
限界を感じたら、レビューや補完のためにCopilotを足すくらいが現実的です。

## 使いすぎで伸びなくなるパターン

避けたい使い方はこのあたりです。

- 生成されたコードを読まずに貼る
- 動いた理由を確認しない
- 変更差分を見ない
- 1つのエラーに対して何度も大きな修正を頼む
- 何を作るか決めずにツールだけ触る

AIに頼るのはアリです。むしろ、頼っていいです。
ただし、毎回最後に差分を見ます。

Cursor実演でも、AIが生成した変更は承認するまで確定ではありません。
「Keep」や「Accept」を押す前に、何が増えたか、何が消えたか、別ファイルを壊していないかを見ます。

さらに安全にするなら、AIに大きな修正を頼む前にGitでコミットします。
これはAIエディタ時代の必須習慣です。壊れても戻れる状態があると、AIに任せるのが怖くなくなります。

## 最初の1週間の使い方

AIエディタを入れたら、最初の1週間は小さなアプリで使います。
おすすめは、タグ生成、文章要約、学習ログ整理のような入力と出力が短いアプリです。

使い方は次の流れです。

1. 作りたいものを1文で書く
2. AIに実装計画だけを出させる
3. 計画を読んで、余計な機能を削る
4. Agentで最小実装を頼む
5. Diffを見てから採用する
6. 動作確認して、壊れたらGitで戻す
7. 詰まったところを記事にする

この流れなら、ツール利用がそのまま学習ログになります。
Wanna Naviでは、AIエディタを「記事を書ける体験」に変えるところまで含めて、開発環境として考えます。

## 料金を見るときの考え方

有料プランを検討する前に、無料枠で1つ作ってみます。
毎日使う、差分確認が楽、エラー解決が早い、という実感が出てから有料化を考える方が安全です。

料金を見るときは、月額だけでなく次の点も見ます。

- 自分の使うOSで安定するか
- 日本語で相談しやすいか
- チームではなく個人利用でも使いやすいか
- 解約やプラン変更が分かりやすいか

最初の目的は、高いツールを選ぶことではありません。
小さなAIアプリを公開するまでの摩擦を減らすことです。
