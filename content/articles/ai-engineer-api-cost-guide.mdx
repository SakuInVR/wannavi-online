AIアプリを作り始めると、かなり早い段階で「API料金が怖い」という不安が出てきます。ボタンを押すたびにお金がかかるのか。バグで無限に呼び出したらどうなるのか。公開したあと、誰かに連打されたら請求が跳ねるのか。

この不安はかなり自然です。AI APIは月額サブスクではなく、基本的には使った量に応じて料金が変わります。実例を整理しても、共通して「料金はリクエスト回数だけでなく、送った文章量、返ってきた文章量、モデルの種類で変わる」と説明されていました。

ただし、最初の個人開発なら怖がりすぎなくて大丈夫です。やることはシンプルで、入力を短くし、出力に上限をつけ、ログを残し、公開前に利用制限を入れる。この4つです。

## API料金は「1回いくら」ではなく「どれだけ読ませて、どれだけ書かせたか」

AI APIの料金を考えるとき、まず覚える単位が「トークン」です。ざっくり言うと、AIが文章を処理するために分けた文字のかたまりです。

料金に効くのは、主にこの3つです。

| 要素 | 何が増えるか | 初心者がやりがちな失敗 |
| --- | --- | --- |
| 入力トークン | AIに読ませる文章量 | 毎回、過去ログや長い説明を全部送る |
| 出力トークン | AIが返す文章量 | 「詳しく」とだけ書いて長文を返させる |
| モデル単価 | 使うAIモデルの価格 | 何でも高性能モデルに投げる |

たとえば、学習ログを要約するだけなら、毎回アプリ全体の説明や過去30日分のログを送る必要はありません。「今日のメモ」と「返してほしい形式」だけで足ります。

公式の料金表は変わるので、具体的な単価は必ず[OpenAI API Pricing](https://openai.com/api/pricing/)の最新表示を確認してください。記事内で大事にしたいのは、単価暗記より「何が増えると請求が増えるか」を理解することです。

## 最初に決める3つの上限

AI APIを使う前に、最低限この3つを決めます。

| 決めること | 最初の目安 | 理由 |
| --- | --- | --- |
| 1日の実行回数 | 自分用なら20〜50回程度 | 試作中の連打を防ぐ |
| 入力文字数 | 1回あたり1000〜2000文字まで | 長文投入で料金と遅延が増える |
| 出力の長さ | 要約なら箇条書き3〜5個 | 不要な長文回答を防ぐ |

最初から完璧な課金管理画面を作る必要はありません。個人開発なら、まずは「自分しか使えない」「1回の入力を制限する」「実行回数をメモする」だけでもかなり安全になります。

## コストを増やすプロンプト、減らすプロンプト

トークン削減の具体策として「毎回同じ長い説明を送らない」「必要な出力形式を短く固定する」「例を入れすぎない」といった考え方が出てきました。

悪い例はこうです。

```text
あなたは優秀なAIアシスタントです。
以下の学習ログを、初心者にもわかるように、できるだけ詳しく、
背景説明も入れて、改善点もたくさん出して、次の計画も細かく作ってください。
```

この書き方は、AIに長い返答を誘導します。最初のアプリでは、もっと短くていいです。

```text
次の学習ログをJSONで整理してください。
返す項目は summary, stuckPoint, nextAction の3つだけ。
各項目は80文字以内。
```

これだけで、出力が短くなり、アプリ側でも扱いやすくなります。AIアプリのコスト管理は、節約というより設計です。何を返してほしいかを狭くすると、料金も失敗も減ります。

## 公開前に入れる安全策

自分だけが使う試作と、公開URLを持つアプリでは危険度が変わります。公開前には、最低限この5つを入れます。

| 安全策 | 何を防ぐか |
| --- | --- |
| 認証 | 誰でもAIを呼べる状態を防ぐ |
| レート制限 | 連打や自動アクセスを防ぐ |
| 入力文字数制限 | 長文投入による高コスト化を防ぐ |
| 出力トークン上限 | AIが長文を返し続けるのを防ぐ |
| エラーログ | 失敗時の再実行ループを見つける |

特に怖いのは、失敗時に自動で何度も再実行する処理です。AI APIの呼び出しが失敗したとき、何も考えずにリトライし続けると、バグなのに料金だけ増えます。最初は「1回失敗したら画面にエラーを出す」くらいの単純な挙動で十分です。

## ログは「料金」だけでなく「機能改善」の材料になる

API利用ログは、請求を怖がるためだけのものではありません。どの機能でAIを呼んだか、入力が長すぎないか、返答が長すぎないかを見ると、アプリの改善点が見えてきます。

最初はこのくらいのログでOKです。

| 項目 | 例 |
| --- | --- |
| 日時 | 2026-05-19 21:30 |
| 機能 | 学習ログ要約 |
| 入力の長さ | 約1200文字 |
| 出力の長さ | 箇条書き3つ |
| 結果 | 成功 / JSON崩れ / タイムアウト |

慣れてきたら、モデル名、処理時間、概算トークン数も入れるとさらに良いです。AIエンジニアとして大事なのは、「動いた」だけで終わらせず、「どれくらいのコストで、どれくらい安定して動いたか」を見られることです。

## 最初に作るなら「料金が読める小さいアプリ」

最初の題材は、料金が読みやすいものにします。おすすめは、1回入力して1回返すだけのツールです。

| アプリ案 | なぜ安全か |
| --- | --- |
| 学習ログ要約 | 1回の入力が短く、出力も固定しやすい |
| 記事タイトル案生成 | 返す数を5個などに制限しやすい |
| ポートフォリオ改善診断 | 自分用から始めやすい |
| タグ生成ツール | JSON出力の練習になる |

逆に、最初からチャットアプリを作ると少し難しくなります。会話履歴を毎回送る必要があり、入力トークンが増えやすいからです。チャットは楽しいですが、コスト管理の練習としては後回しで大丈夫です。

AI API料金の不安は、「見えないものに課金される感じ」が原因です。だから最初にやるべきことは、安いモデルを探すことだけではありません。入力、出力、回数、ログを見えるようにすることです。

それができると、AIアプリ開発はかなり安心して試せます。
