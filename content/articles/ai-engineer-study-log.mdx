AIエンジニアを目指す人の学習ログは、「今日はLangChainを触った」「RAGを勉強した」だけだと、読者にも未来の自分にも残りにくいです。

収益につながる記事に変えるなら、学習ログを「失敗した人が次に進める手順」に変換します。読む人が知りたいのは、あなたが何時間勉強したかではなく、「どこで詰まり、何を変えたら動いたのか」です。

YouTubeでAIポートフォリオとAIエンジニアロードマップ系の動画を見比べると、共通しているのはここでした。講座を増やすより、動くものを作る。作ったものを説明できる形にする。学習ログは、その説明力を育てる素材です。

## 学習ログは5項目で書く

まず、毎回ゼロから文章を考えないように、型を固定します。

| 項目 | 書くこと |
| --- | --- |
| 作ろうとしたもの | 例: PDF質問回答アプリ、家計簿分類AI、議事録要約ツール |
| 詰まったところ | 例: PDFから表が崩れる、API料金が読めない、回答が毎回ぶれる |
| 原因の仮説 | 例: chunkサイズが大きい、プロンプトが曖昧、評価データがない |
| 試した修正 | 例: chunkを小さくする、Pydanticで出力形式を固定する、テスト質問を作る |
| 次に使う人へのメモ | 同じ失敗を避けるための手順 |

この型にすると、「勉強しました」ではなく「問題を切り分けました」という記事になります。AIエンジニアに必要なのは、モデル名をたくさん知ることだけではなく、動かないシステムを観察して直す力です。

## 日記を記事に変える例

たとえば、学習ログとしてこう書いたとします。

```md
今日はRAGを試した。
PDFを読み込ませたけど、回答が微妙だった。
LangChainが難しかった。
```

このままだと、読者は何も持ち帰れません。記事にするなら、こう変えます。

```md
## やりたかったこと
自分のメモPDFに質問できる小さなRAGアプリを作る。

## 詰まったこと
PDF内の表を読み込ませると、回答が抜けたり、別の項目と混ざったりした。

## 原因の仮説
PDFのテキスト抽出時に、表の行と列の関係が崩れている可能性がある。

## 試したこと
chunkサイズを小さくし、質問を3パターン用意して回答を比較した。

## 次にやる人へのメモ
PDF RAGは、モデル選びより先に「PDFから正しくテキストが取れているか」を確認した方がいい。
```

これなら、同じところで詰まった人に届きます。検索されるのは「RAG 勉強した」ではなく、「RAG PDF 表 回答 おかしい」「LangChain PDF 抽出 崩れる」みたいな具体的な困りごとです。

## ポートフォリオになるログの条件

AIポートフォリオ系の動画で強調されていたのは、「作った」だけでは弱いという点です。強いポートフォリオは、次の3つがそろっています。

| 要素 | 具体例 |
| --- | --- |
| 動くもの | Streamlit、Next.js、FastAPIなどで触れる形にする |
| コード | GitHubで読めるようにする |
| 説明 | なぜその設計にしたか、どこで失敗したかを書く |

たとえば「家計簿CSVを分類するAI」を作ったなら、記事ではモデル名より先に、こういう話を書く方が読者に刺さります。

- 食費と日用品の分類が混ざった
- 店名だけでは分類できなかった
- ルールベースとAI判定を組み合わせた
- ローカルLLMだと精度が足りない場面があった
- APIを使う場合はコスト上限を決めた

これはただの制作メモではありません。次に同じアプリを作る人にとって、かなり価値のある失敗例です。

## 収益導線は自然に置く

学習ログ記事でアフィリエイトを置くなら、無理に売り込むより「この段階で必要になるもの」を置く方が自然です。

| 記事の場面 | 置きやすい導線 |
| --- | --- |
| API料金で悩んだ | APIコスト管理、クラウドクレジット、学習講座 |
| デモ公開したい | Vercel、クラウド、ドメイン、GitHub関連 |
| ローカルLLMを試した | GPU、メモリ、ストレージ、ミニPC |
| ポートフォリオを整えたい | 技術書、講座、転職・学習サービス |

大事なのは、読者が「今それ欲しいかも」と思う位置に置くことです。たとえば、RAG記事の最後にいきなり高額スクールを貼るより、「まずは小さく試すなら、この講座・この本・この環境がある」と置く方が信頼されます。

## 1記事にする判断基準

学習ログを全部記事にする必要はありません。記事にする価値が高いのは、次のどれかに当てはまるログです。

- エラーや失敗の原因がはっきりした
- ツール選びで迷った
- 料金や制限で判断が必要だった
- デモ公開まで進んだ
- 他の初心者も同じところで詰まりそうだった

逆に、「動画を見た」「ドキュメントを読んだ」だけの日は、まだ記事にしなくて大丈夫です。その日はメモに残して、実装で詰まった日に記事化します。

## 今日からの書き方

今日から学習ログを書くなら、1日1本の完璧な記事を狙わなくていいです。まずは次の形で、短いメモを残します。

```md
今日作ったもの:
詰まったところ:
原因っぽいもの:
試した修正:
まだ残っている問題:
記事にするなら狙う検索語:
```

このメモが3つ、5つ、10個とたまると、ただの勉強記録ではなく、AIエンジニアになるまでのロードマップになります。しかも、同じ道を通る人が検索でたどり着ける記事になります。

学習ログは、自分の弱さを公開するものではありません。次の読者の時間を短縮するための、かなり強いコンテンツです。
