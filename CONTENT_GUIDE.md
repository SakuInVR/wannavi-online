# Wanna Navi Content Guide

Wanna Naviの記事を書くときの、カテゴリ、収益導線、監査条件の考え方です。

## 記事の役割

記事は大きく3種類に分けます。

```text
集客記事:
検索から読者を集める記事。
例: AIエンジニアになる最初の1ヶ月、DTMで4小節ループを作る方法

悩み解決記事:
読者のつまずきを解決する記事。
例: VRChatのシェーダーでつまずく理由、プラグインを買う前に確認すること

収益記事:
買い物、登録、比較、教材、機材などに近い記事。
例: 最初の機材と買う順番、ヘッドホン選び、開発ツール
```

収益記事は「読者がお金を使う直前に読む記事」です。
押し売りではなく、選ぶ理由、買わなくていいもの、失敗しにくい順番を整理します。

## affiliateIntent

`affiliateIntent` は、このサイト内で使う管理ラベルです。
Googleや広告サービスの正式用語ではありません。

```text
low:
読み物、体験談、思想、ゆるい学習ログ寄り。
収益リンクは弱め。

medium:
やり方、ロードマップ、道具紹介に自然につながる記事。

high:
おすすめ機材、教材、ツール、比較、買う順番など、収益リンクに近い記事。
```

迷ったら、この記事を読んだ人が次に何をしたくなるかで決めます。

```text
もっと知りたい        -> low
試してみたい          -> medium
買う/登録する候補を探す -> high
```

## 監査条件

`npm run audit:site` は、サイトの作り忘れを防ぐためのチェックです。
人間の判断を置き換えるものではなく、「最低限の土台が壊れていないか」を確認します。

現在の主なチェック:

```text
必須ページがある
必須コンポーネントがある
各カテゴリに4記事以上ある
各カテゴリにaffiliateIntent: "high" の記事が1本以上ある
記事ページにPR表記、広告枠、カテゴリCTAが自動で入る
sitemap、robots、RSS、ads.txt がある
Analytics、AdSense、Search Console用の設定口がある
CIで検証が走る
```

## 記事を書くときの目安

各カテゴリでまずそろえたい構成:

```text
1本: 最初のロードマップ
1本: よくあるつまずき解決
1本: 実践チュートリアル
1本: 機材/教材/ツール選び
```

このうち、機材/教材/ツール選びの記事は `affiliateIntent: "high"` にしやすいです。

## カテゴリを追加するとき

カテゴリを追加するときは、まず `src/lib/site.ts` の `categories` に1件足します。
カテゴリページ、トップページ、サイトマップ、記事レポート、監査はこのカテゴリ一覧を参照します。

追加直後は、次をそろえるとサイト全体の品質を保ちやすくなります。

```text
4記事以上
affiliateIntent: "high" の記事を1本以上
カテゴリ別の収益CTA
外部リンクID
```

収益CTAは `src/lib/monetization.ts`、外部リンクIDと環境変数名は `src/lib/outbound-links.ts` に追加します。
ASPやAmazonの実リンクは、コードではなくVercel Environment Variablesの `AFFILIATE_*_URL` に入れます。

## draft

新しく作った記事は `draft: true` から始めます。
書きかけ記事はサイト、RSS、sitemap、タグページ、カテゴリページに出ません。
直URLでも表示されないように、記事ページ側でも404扱いにします。

公開するときはfrontmatterを次のように変更します。

```yaml
draft: false
```

コマンドで公開状態にすることもできます。

```bash
npm run publish:article -- article-slug
```

このコマンドは `draft: false` と `updatedAt` を自動で更新します。

公開を取り下げる場合:

```bash
npm run unpublish:article -- article-slug
```

取り下げ時も `draft: true` と `updatedAt` を自動で更新します。

リライト後に公開状態は変えず、更新日だけ反映する場合:

```bash
npm run touch:article -- article-slug
```

`draft: true` の記事は、収益記事として扱わないため `affiliateIntent: "high"` にしないでください。

## 公開記事の最低品質

`npm run validate:content` は、公開記事に対して最低限の本文量と見出し数を確認します。

```text
本文量: 空白を除いて220文字以上
見出し: ## 見出しが2つ以上
```

これは良い記事を保証するものではなく、薄すぎる記事をうっかり公開しないための安全柵です。

## 収益導線の置き方

記事ページには自動で以下が入ります。

```text
PR表記
カテゴリ別CTA
記事下広告枠
関連記事
```

本文内で追加したい場合だけ、MDXに次を置きます。

```mdx
<ToolRecommendation
  name="おすすめ道具名"
  reason="なぜ必要か。初心者にどう効くか。"
  priceHint="無料枠からでOK"
/>
```

```mdx
<AffiliateCTA
  title="関連する教材・機材・サービス"
  description="押し売りではなく、次の一歩に必要な選択肢として紹介する。"
  label="候補を見る"
/>
```

## 大事な方針

収益化を狙っても、記事の主役は読者の「なりたい」です。
紹介リンクは、読者が次に進むための選択肢として置きます。

## ユーザーとCodexの記事分担

記事運用は、次の分担で進めます。

```text
ユーザー:
見たい記事テーマ、体験談、気になる商品やジャンルを出す

Codex:
MDX記事を書く
frontmatterを整える
収益導線を自然に置く
preflightで検証する
GitHubへpushして本番反映を確認する
```

ユーザーからの記事依頼は、短くてOKです。

```text
「大人からピアノを始めたい人の記事」
「VRChatで衣装改変したい人向け」
「AIエディタの選び方」
```

情報が足りない場合は、Codex側で読者像、検索意図、記事構成を補って書きます。

## YouTube動画を使った記事作成

今後の記事は、原則として [VIDEO_RESEARCH_WORKFLOW.md](./VIDEO_RESEARCH_WORKFLOW.md) の順番で作成します。

```text
タイトル決定
↓
YouTube動画3本選定
↓
Gemini APIまたはユーザー指定AIで分析
↓
具体例、つまずき、判断基準を抽出
↓
MDX記事化
↓
検証して公開
```

動画を参照する目的は、記事に実際の練習・作業・失敗例の手触りを入れるためです。
動画の内容を丸写しせず、複数動画から共通する読者の困りごとと実践手順を抽出します。
