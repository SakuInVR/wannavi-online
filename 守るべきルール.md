# Wannavi.online LLM Handoff

この文書は、Codex以外のLLMに `wannavi.online` の記事執筆、ASP連携、品質改善、自動化パイプライン運用を引き継ぐための運用仕様です。

最重要ゴールは「記事を増やすこと」ではありません。ユーザーが見たい記事テーマを出し、LLMが調査、構成、執筆、広告候補整理、検証、改善キュー作成まで進めることで、記事をアップしたあとに継続的に収益化と改善が回る仕組みを作ることです。

## サイトの前提

- サイト名: Wanna Navi
- ドメイン: `https://www.wannavi.online`
- コンセプト: 「〇〇になりたい」人のための実践ロードマップ型メディア
- 主要カテゴリ:
  - `ai-engineer`: AIエンジニアになりたい
  - `dtm`: DTMerになりたい
  - `vr-creator`: VRクリエイターになりたい
  - `instrument-player`: 楽器演奏者になりたい
  - `video-creator`: 動画クリエイターになりたい
- 現在の実装: Next.js + MDX記事 + JSON管理ファイル
- 将来方針: Supabaseを中核にした記事、調査、広告、公開、改善状態の永続化

## LLMの役割分担

ユーザーは「見たい記事」「扱いたい商品」「気になる方向性」を出す係です。

LLMは以下を担当します。

1. 記事テーマを検索意図に落とし込む
2. タイトル案を決める
3. YouTube動画3本を選定する
4. 動画内容をGemini APIまたは指定AIで分析する
5. 公式情報、商品情報、ASP条件を確認する
6. 具体的な読者シーン、判断基準、失敗例を入れて記事を書く
7. MDX frontmatterと本文を整える
8. ASP広告、商品、画像広告の候補を記事意図に合わせて割り当てる
9. preflight、build、本番確認まで進める
10. Search Console、GA4、ASP成果などから改善キューを更新する

提案だけで止めないでください。実装、検証、次の改善キュー作成まで進めます。

## 絶対に守る編集ルール

- 既存のユーザー変更を勝手に戻さない
- ASPリンクや広告URLを捏造しない
- 承認されていないASP案件を「承認済み」として扱わない
- 生成AIの出力をそのまま公開しない
- YouTubeの内容を丸写ししない
- 記事本文に「動画を見たところ」「YouTubeでは」などの制作裏事情を書かない
- 読者向けページに `affiliateIntent`, `reviewed`, `monetized`, `high` などの内部状態を表示しない
- 文字化け、プレースホルダー、空リンク、未検証の断定がある記事を公開しない
- `content/deployment-checks.json` は生成物なのでコミットしない

## 記事を書く基本プロセス

### 1. タイトル決定

タイトルは「誰が」「何を判断できるようになるか」を含めます。

良い例:

- DTM初心者がオーディオインターフェースを買う前に確認すること
- VRChat向けゲーミングPCはどのスペックから選べばいいか
- 大人からピアノを始める人の最初の1か月練習ルーティン

悪い例:

- 初心者向け完全ガイド
- おすすめまとめ
- 失敗しない方法

抽象タイトルは避け、検索者の悩みと次の行動が見えるタイトルにします。

### 2. YouTube動画3本の選定

記事を書く前に、タイトルに合うYouTube動画を3本選びます。

役割の目安:

1. 全体像を説明する初心者向け動画
2. 実演、練習、作業画面、比較などがある動画
3. 失敗談、購入判断、経験者視点、比較視点がある動画

同じ主張の動画だけを3本並べないでください。読者がつまずく場所、買うべきもの、まだ買わなくていいものを判断するために選びます。

### 3. 動画分析

動画はGemini APIまたはユーザー指定AIで分析します。

コマンド例:

```bash
npm run analyze:youtube -- "article-slug-or-title" "https://www.youtube.com/watch?v=..." "https://www.youtube.com/watch?v=..." "https://www.youtube.com/watch?v=..."
```

分析結果は `research/youtube/` にJSONとして保存します。

動画分析から抽出するもの:

- 初心者が実際に止まりやすい場面
- 最初にやるべき小さな手順
- 買っていいもの、まだ買わなくていいもの
- 失敗しやすい判断
- 必要な機材、教材、サービス
- 記事に入れるべき具体例

重要: 動画を根拠にしても、記事本文はWanna Navi自身の編集記事として書きます。

### 4. 公式情報と商品情報の確認

YouTube 3本だけでは一次情報として弱いです。特に収益意図が高い記事では、最低1つは以下を確認します。

- 公式サイト
- 製品仕様
- 価格、対応OS、必要スペック
- スクールやサービスの受講条件
- ASP案件の成果条件、禁止事項、使用可能素材

記事の断定は、この確認範囲を超えないようにします。

### 5. MDX記事化

記事は `content/articles/<slug>.mdx` に作成します。

frontmatterの基本形:

```yaml
---
title: "記事タイトル"
description: "検索結果に出ても意味が伝わる説明文"
category: "dtm"
publishedAt: "2026-05-20"
updatedAt: "2026-05-20"
tags:
  - "DTM"
  - "初心者"
affiliateIntent: "high"
sourceVideos:
  - "https://www.youtube.com/watch?v=..."
  - "https://www.youtube.com/watch?v=..."
  - "https://www.youtube.com/watch?v=..."
draft: true
---
```

新規記事は原則 `draft: true` で開始します。公開できる品質になってから `npm run publish:article -- <slug>` を使います。

## 記事品質ルール

公開記事には以下を必ず入れます。

- 読者の具体的な開始シーン
- 最初にやること
- つまずきやすいポイント
- 判断基準
- 買う前に確認する条件
- 買わなくていい条件
- 表、チェックリスト、比較表、手順表のどれか
- 今日できる1ステップ
- 参考にした動画3本のURL

文章の方向性:

- 読者に近い言葉で書く
- でも専門的な判断は曖昧にしない
- 「おすすめです」だけで終わらせない
- なぜそれを選ぶのか、誰には不要なのかを書く
- 収益導線より先に、読者の判断を助ける

避ける文章:

- 「継続が大事です」だけの精神論
- 「まずは基礎を学びましょう」だけで終わる説明
- 「動画を見たところ」という制作メモ
- 商品を押すだけの文章
- 誰にでも当てはまる一般論

## affiliateIntentの意味

`affiliateIntent` はサイト内部の管理用ラベルです。読者には表示しません。

- `low`: 体験談、考え方、ゆるい学習ログ
- `medium`: 始め方、ロードマップ、ツール紹介に自然につながる記事
- `high`: 機材、教材、スクール、サービス、比較、購入判断に近い記事

`high` の記事は、公開前に最低1つの広告または商品候補が必要です。

## ASPと広告素材の扱い

ASPの主要候補:

- A8.net
- もしもアフィリエイト
- ValueCommerce

ASP作業では、ブラウザ操作やログイン状態に依存しすぎないよう、結果を必ず記録します。

記録する項目:

- ASP名
- 広告主名
- 案件名
- 提携状態: `candidate`, `applied`, `approved`, `rejected`, `expired`
- 申請日、承認日
- クリックURL
- 画像URL
- 画像alt
- 対象記事slug
- なぜこの記事に合うのか
- 禁止事項や注意点

商品・広告の管理ファイル:

- `content/affiliate-products.json`
- `content/ad-creatives.json`
- `content/affiliate-program-snapshots.json`
- `content/asp-opportunities.json`

画像広告を優先します。ただし、読者体験を壊す大きすぎる広告、記事内容とズレる広告、承認状態が不明な広告は使いません。

## 収益導線の美学

このサイトでは、不自然な「PR枠」や読者の集中を切る長い広告説明は避けます。

望ましい形:

- 記事の判断表の直後に、文脈に合う商品カードを置く
- 「この条件に当てはまる人向け」という形で広告を置く
- 画像、商品名、用途、向いている人、注意点をセットで出す
- リンクは `rel="sponsored"` 相当の扱いにする

避ける形:

- 記事冒頭に長い広告宣言ブロックを置く
- 「収益導線: high」のような内部情報を見せる
- 記事内容と関係ない広告を貼る
- クリックさせるためだけの強い煽り文を書く

## 自動化パイプライン

目標状態は以下です。

```text
idea -> researched -> drafted -> reviewed -> monetized -> ready -> published -> verified -> improving
```

状態の意味:

- `idea`: 記事テーマがある
- `researched`: 動画、公式情報、商品情報の調査が済んでいる
- `drafted`: MDX下書きがある
- `reviewed`: 記事品質ゲートを通った
- `monetized`: 広告または商品候補が対応している
- `ready`: 公開前チェックを通せる
- `published`: 本番公開対象になった
- `verified`: 本番URLで確認済み
- `improving`: Search ConsoleやASP結果から改善対象になっている

JSON管理ファイル:

- `content/article-states.json`
- `content/automation-pipeline.json`
- `content/improvement-tasks.json`
- `content/automation-backlog.json`

DB化する場合の将来テーブルは `supabase/migrations/202605200001_affiliate_automation_control_plane.sql` を参照します。

## よく使うコマンド

記事作成:

```bash
npm run new:article -- <slug> "記事タイトル" <category>
```

商品起点の記事作成:

```bash
npm run new:product-article -- <product-id> "記事タイトル"
```

公開:

```bash
npm run publish:article -- <slug>
```

非公開:

```bash
npm run unpublish:article -- <slug>
```

動画分析チェック:

```bash
npm run validate:video-research
```

パイプライン同期:

```bash
npm run pipeline:sync
npm run asp:sync
npm run improvements:sync
```

同期コマンドは依存関係があるので、この順番で実行します。並列実行しないでください。

総合チェック:

```bash
npm run preflight
```

本番確認:

```bash
npm run production:verify
```

改善キュー確認:

```bash
npm run report:improvements
```

自動化監査:

```bash
npm run automation:audit
```

## 公開前チェックリスト

公開前に必ず確認します。

- `draft: false` にする理由がある
- titleとdescriptionが検索意図に合っている
- sourceVideosが3本ある
- `research/youtube/` に対応する分析JSONがある
- 本文に具体的な場面、手順、判断基準がある
- 表またはチェックリストがある
- 「動画を見たところ」など制作裏事情が本文にない
- high intent記事には商品または広告候補がある
- 内部管理ラベルが読者に表示されない
- `npm run preflight` が通る
- 本番反映後に `npm run production:verify` を通す

## 本番運用上の注意

- 現在、本番サイトはすでに公開されています。壊れた記事や文字化けをpushするとそのまま読者に見えます。
- Vercelの正しい本番プロジェクトは `wannavi_online` です。
- 重複プロジェクトがある可能性があります。削除や連携解除はユーザー確認なしに行わないでください。
- Git pushは公開完了ではありません。本番URLでの確認までが公開完了です。
- `content/deployment-checks.json` は現在のcommit SHAを含む生成物です。コミットすると自己参照ループを起こすため、Git管理しません。

## 現在の既知リスク

- 既存の一部ドキュメント、MDX、JSONに文字化けが残っています。
- Supabase schemaはありますが、現状のサイトはまだDB駆動ではなく、MDXとJSONが中心です。
- ASP操作はブラウザログイン状態に依存しやすく、再現性が弱いです。
- high intent記事の一部は、まだ記事意図に完全一致するASP商品が不足しています。
- 広告画像素材は承認状態、利用条件、文脈一致を確認してから配置する必要があります。

## 別LLMへの最初の指示例

以下のように依頼すると、作業がぶれにくいです。

```text
あなたはWannavi.onlineの記事制作と収益化パイプラインを担当するLLMです。
LLM_HANDOFF.mdを最上位ルールとして読み、記事テーマに対して次を実行してください。

1. タイトルを決める
2. YouTube動画3本を選ぶ
3. Gemini APIまたは指定AIで動画を分析する
4. 公式情報とASP商品候補を確認する
5. MDX記事を書く
6. affiliateIntentと商品/広告候補を整える
7. preflightを通す
8. 本番反映後にproduction verifyを行う
9. 改善キューを更新する

記事本文は読者向けの自然なブログ記事として書き、動画参照の制作過程や内部管理ラベルを本文に出さないでください。
```

## 判断に迷ったときの優先順位

1. 読者が次に何をすればいいか明確か
2. 記事に具体的な経験例、判断基準、失敗例があるか
3. 収益導線が記事意図と自然につながっているか
4. ASPや広告素材の状態が記録されているか
5. 本番で壊れていないか

記事数より、信頼できる記事と改善ループを優先してください。
