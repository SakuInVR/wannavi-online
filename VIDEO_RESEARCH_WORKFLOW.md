# Video Research Workflow

Wanna Naviの記事は、今後すべて次の順番で作成します。

```text
タイトルを決める
→ タイトルに合うYouTube解説動画を3本選ぶ
→ Gemini APIまたはユーザー指定AIで3本を分析する
→ 分析結果から具体例、つまずき、判断基準、買う/買わない基準を抜き出す
→ MDX記事として書く
→ 検証して公開する
```

## なぜ動画を参照するか

Wanna Naviの記事で一番避けたいのは、一般論だけで終わることです。

読者は「なりたい」と思って検索しています。必要なのは、ふわっとした励ましではなく、実際の作業画面、練習の止まり方、機材選びの迷い、失敗しやすい判断を見たうえでの道案内です。

YouTubeの解説動画を3本見ることで、記事に次の要素を入れます。

- 実際の作業や練習の場面
- 初心者が止まりやすい瞬間
- 最初にやるべき小さな手順
- 買うべきもの、まだ買わなくていいもの
- 今日できる1ステップ
- 収益導線につながる自然な判断基準

動画の内容を丸写しするのではなく、複数動画に共通する実践知を、Wanna Naviの読者向けに再構成します。

## 記事作成の順番

1. 記事タイトルを決める
2. タイトルに合うYouTube動画を3本選ぶ
3. `npm run analyze:youtube` で3本をGemini分析する
4. `research/youtube/` に保存されたJSONを読む
5. MDX frontmatterに `sourceVideos` を3本入れる
6. 本文末尾に `## 参考にした視点` を入れる
7. 動画由来の具体例を、読者向けの手順や判断基準に変換する
8. `npm run preflight` を通す
9. GitHubへpushする
10. 必要に応じて本番URLで表示を確認する

## Geminiで動画分析する

```bash
npm run analyze:youtube -- "article-slug-or-title" "https://www.youtube.com/watch?v=..." "https://www.youtube.com/watch?v=..." "https://www.youtube.com/watch?v=..."
```

`GEMINI_API_KEY` が必要です。分析結果は `research/youtube/` にJSONとして保存されます。

PowerShellで日本語引数が文字化けする場合は、分析ファイル名を安定させるために英字slugを使います。記事本文とfrontmatterのタイトルは日本語で書きます。

## 動画選定基準

3本は、できるだけ役割を分けます。

```text
1本目: 初心者向けの全体像
2本目: 実演、練習、作業画面があるもの
3本目: 買い物、失敗談、比較、経験談があるもの
```

同じ主張の動画だけを3本選ばないようにします。記事では、複数動画に共通する内容を中心にしつつ、1本だけの主張は「一例」として扱います。

## 記事に必ず入れること

- 動画から見えた具体的な場面
- 初心者が止まりやすい瞬間
- 最初にやる小さな手順
- 買わなくていいもの、急がなくていいもの
- 必要なら収益導線
- 今日できる1ステップ
- 参考にした動画3本のURL

## 文章の品質基準

次の文章は、公開記事に残さないでください。

- 「まずは基礎を学びましょう」だけで終わる説明
- 「継続が大事です」だけで終わる締め
- 具体的な作業、練習、画面、判断が出てこない段落
- 読者が何を買うべきか、まだ買わなくていいか分からない導線
- 動画を見なくても書ける一般論だけの本文

記事は必ず「読者が今日なにをすればいいか」まで落とします。

## 注意点

- 動画内容を長く引用しない
- 動画制作者の主張を絶対視しない
- 商品紹介を動画の影響だけで決めない
- 公開記事では参照したYouTube URLを明記する
- Geminiの分析結果は事実確認なしで鵜呑みにしない

## 検証

動画分析型の記事は、次を満たす必要があります。

```text
sourceVideos が3本ある
対応する research/youtube の分析JSONがある
本文に ## 参考にした視点 がある
```

このチェックは `npm run validate:video-research` と `npm run preflight` に含まれます。
